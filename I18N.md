# `@rhythm-archive/bms-editor` — i18n Contract

This package exposes user-visible strings (toolbar labels, panel headers,
dialogs) through an injected translator so consumers can pick their own i18n
runtime. The package itself never depends on a specific i18n library.

## Quick start

```tsx
import { I18nProvider, NoteChartEditor, type Translator } from '@rhythm-archive/bms-editor';
import { useTranslation } from 'react-i18next'; // your runtime, optional

function App() {
  const { t, i18n } = useTranslation('bms-editor');
  const provider = useMemo(
    () => ({ t: t as Translator, locale: i18n.language }),
    [t, i18n.language],
  );
  return (
    <I18nProvider value={provider}>
      <NoteChartEditor /* ... */ />
    </I18nProvider>
  );
}
```

The `<I18nProvider>` wraps the editor once; every internal component reaches
the translator via `useI18n()` — no prop drilling.

## What you must translate

The full key surface lives in [`src/i18n/defaults.ts`](./src/i18n/defaults.ts).
At a glance:

| Group                             | Approx keys |
|-----------------------------------|-------------|
| `toolbar.*`                       | 10          |
| `panels.keysound.*`               | 8           |
| `panels.header.*`                 | 5           |
| `panels.minimap.*`                | 1           |
| `panels.noteInfo.*`               | 2           |
| `panels.statusBar.*`              | 3           |
| `panels.noteSearch.*`             | 3           |
| `contextMenu.*`                   | 5           |

Phase 2 of the i18n roadmap will grow this surface to ~1,000 keys. The
`BmsEditorI18nKey` type is derived from `defaultMessages`, so consumers who
type their own translator as `Translator` get compile-time misspell checks.

## What happens if you skip `<I18nProvider>`

`useI18n()` returns a fallback that resolves keys against `defaultMessages`
(English). The library stays usable; you simply lose translation support.

## Interpolation

Keys with interpolation (e.g., `{{count}} notes`) follow the i18next
convention: pass `params` as the second argument to `t`. The bms-editor
package does **not** interpret the params — it forwards them to your `t`
runtime.

```ts
t('panels.statusBar.notes', { count: 42 });
```

## Locale change behaviour

Components subscribe to the React Context value. When the consumer's
`<I18nProvider value={...}>` receives a new `t` reference (e.g., after
`i18n.changeLanguage()`), every internal component re-renders with the new
translation. No imperative API; no manual flush.

If your translator memoizes badly and the `t` reference is stable across
language changes (common in some i18n runtimes), you must also pass `locale`
in the provider value so the memo invalidates.

## Out of scope

- **User chart data** (`#TITLE`, `#ARTIST`, keysound filenames) — never
  translated by this package; those values are user input and pass through
  unchanged.
- **Number/date formatting** — handled by the consumer via `Intl.*`.
- **RTL layouts** — not supported.

## Versioning

Adding a key to `defaultMessages` is a **minor** version bump. Removing or
renaming a key is a **major** version bump (breaks consumer translation
bundles).
